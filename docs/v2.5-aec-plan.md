# v2.5.0 — OS-level echo cancellation (the plan)

*Written 2026-07-20, the morning after the v2.3/v2.4 field failures. This
lives in the PRIVATE dev repo; the public line sits on v2.2.0 until this
survives the same field tests that killed its predecessors.*

## The one-sentence goal

Talk over the agent on open speakers, in a normal voice, at any playback
volume — by letting macOS subtract the Mac's own audio from the mic at the
driver level instead of guessing in userspace.

## Why every previous attempt died (read before building)

Three features, one wall. All field-tested on a MacBook Pro, scores logged.

| Attempt | Idea | Cause of death |
|---|---|---|
| v2.3 voice-lock | mic answers only the enrolled voice; its own TTS fails the check | same-speaker embedding scores scatter wider across utterances (0.17–0.63 live) than the gap to a different speaker (a video scored 0.24) — not separable on a laptop mic |
| v2.4 echo gate (residual) | envelope-match the mic against the known playback; unexplained energy = person | echo-rejection works (0 false cuts, 2 sessions, even max volume) — but a couch-voice talk-over is a few % of the mic signal next to speakers inches away: foreign scores 0.15–0.47 vs echo 0.10–0.47. Full overlap. |
| v2.4 low-band channel | human bass (85–155Hz) that laptop speakers "can't emit" | MacBook Pro woofers emit it fine: echo low-band 0.16–0.31 vs talk-over ~0.26. Overlap again. |

The pattern: **any scalar feature computed from a mic drowning in its own
speaker output produces overlapping distributions.** The signal has to be
cleaned before feature extraction — which is exactly what an acoustic echo
canceller does, and macOS ships a battle-tested one (AUVoiceIO, the
voice-processing unit behind FaceTime).

What survives from v2.4 regardless (not wasted): the echo-transcript
backstop, the never-cancel-work resend, the collection deadline, the
no-PortAudio-calls-at-cut contract, the wall-clock EchoRef, cut/score
telemetry in transcripts, and the residual gate itself — which becomes the
FALLBACK layer wherever native AEC isn't available.

## The design

### Capture path (the heart of it)

`VoiceProcessingMic` — same interface as `audio.Mic` (start / stop /
frames / set_muted), backed by AVAudioEngine with voice processing:

- `engine.inputNode.setVoiceProcessingEnabled_(True)` (macOS 10.15+),
  via PyObjC (`pyobjc-framework-AVFoundation`).
- `installTapOnBus_` delivers AVAudioPCMBuffers on a realtime thread →
  convert to float32 mono 16k (AVAudioConverter; VPIO dictates its own
  native rate, typically 24k or 48k) → `loop.call_soon_threadsafe` onto
  the same bounded asyncio queue the PortAudio mic uses today. Tap block
  stays allocation-light; the conversion may need to happen on the
  consumer side if the tap thread can't afford it.
- AGC / noise-suppression toggles (`isVoiceProcessingAGCEnabled` etc.)
  surfaced as config later; defaults first.

### Playback path (decided by the spike, not by hope)

Open question with two answers, each fine:

- **(a) VPIO cancels all device output** (it taps the hardware path, like
  iOS): keep the current PortAudio Speaker untouched. Smallest diff.
- **(b) VPIO only cancels audio played through its own paired output
  unit**: route TTS through an AVAudioEngine playerNode
  (`VoiceProcessingSpeaker`: scheduleBuffer + stop()). Bonus if so:
  playerNode.stop() is instant and safe — the entire CoreAudio
  cut-deadlock class from 2026-07-19 disappears with PortAudio output.

### Ducking

macOS ducks other audio when VPIO detects near-end speech; macOS 14 adds
`AVAudioVoiceProcessingOtherAudioDuckingConfiguration` to control it.
Ducking is arguably a FEATURE here (it quiets itself when you start
talking — exactly the UX we want) but it must not fight the barge cut.
Observe in the spike; configure deliberately in Phase 2.

### Mode resolution (what users get, no flags needed)

```
speakers + macOS + engine works  → native AEC (this build)
speakers + native AEC failed     → echo-gate residual (v2.4 fallback, proven zero-false-cut)
speakers + --no-aec              → gaps-only mute-while-speaking (v2.2 behavior)
headphones                       → plain full duplex (unchanged)
phone mode                       → unchanged (phone's own AEC)
Linux                            → echo-gate fallback (no VPIO)
```

Card copy says which layer armed. `--aec` grows values: `native` (default
resolution), `gate`, `off`.

### Orchestrator changes when native AEC is armed

- `echo_gate=None` (nothing to gate — the mic is clean), barge onset back
  to 450ms, double-confirm off.
- KEEP: silero speech-check, the echo-transcript backstop (belt against
  imperfect cancellation — AEC leaves residual echo at max volume
  sometimes), never-cancel-work, collection deadline.
- TV/roommate speech WILL still barge — same as every version; that's
  voice-lock's unsolved job (see "explicitly not in v2.5").

## Phase 0 results (2026-07-20, MacBook Pro mic + speakers, macOS 26.1) — GO

The dB gate was measuring the wrong thing; the product gate passed decisively.

- **Raw attenuation looked weak** (~4dB mean vs the ~20dB bar): VPIO's AGC
  pumps gain into the post-AEC residual, so RMS-above-floor understates how
  dead the echo actually is. Per-second profiles oscillate between −60 and
  −24dBFS — energy survives, speech does not.
- **The echo is unintelligible after VPIO.** whisper base.en transcribes the
  baseline capture verbatim (the entire self-barge problem in one line) and
  returns NOTHING from either VPIO capture. Silero rejects the five loudest
  residual windows in both configs (baseline: 5/5 "speech"). Echo can no
  longer become a turn, a transcript, or a confirmed barge — the v2.3/v2.4
  failure class dies at the driver.
- **Question (a)/(b) answered: (a).** VPIO references ALL device output at
  the HAL — PortAudio-played TTS is cancelled identically to engine
  playerNode playback (both STT-invisible). The PortAudio Speaker stays
  untouched; native mode is a capture-side change only.
- **Tahoe quirk:** enabling voice processing flips the input node to
  9ch/48kHz deinterleaved; all channels are identical copies — take ch0,
  decimate 48k→16k.
- **Stability:** 6/6 enable-capture-teardown cycles clean WITH full teardown
  (`setVoiceProcessingEnabled_error_(False)` + `engine.reset()` after stop).
  Skipping those wedged the input into digital silence (−180dBFS) after ~4
  cycles in one process — the product keeps ONE engine per process and
  reuses it across Ctrl-T relaunches.
- CPU during capture+playback: ~10% of one core. Ducking: not observed on
  PortAudio playback at normal volume.

## Field round 1 (2026-07-20 ~9 PM, Nate live) — protocol passed, 3 complaints, all fixed

Talk-over, barge cuts, pause/wake mid-task, Ctrl-C: worked. Complaints:
slow pause/wake reaction, onset-rejected log spam, one turn stuck
"listening" 2-3 minutes. One root cause dominated, plus one regression:

- **webrtc VAD misreads the voice-processed signal.** Measured: 60-76% of
  pure idle-silence frames flagged "voiced" (any aggressiveness, AGC on or
  off — AGC was ruled OUT by the same A/B). The NS residual hiss is
  speech-shaped to webrtc, so trailing-silence windows kept resetting:
  every turn-end in Nate's --debug log counted 2-3× his real speaking time,
  and one turn rode the 90s max_utterance ceiling — the "stuck for
  minutes". The energy VAD on the same captures: 0% idle, 2% post-TTS, 3%
  during-echo (NS flattens the floor to a steady -49dBFS — ideal for a
  threshold). Fix: native mode swaps webrtc → energy at the ladder with a
  printed note; energy is already the default.
- **Commands paid the full 1200ms silence wait.** Fix: adaptive close —
  speech totals ≤1.5s (command-shaped: "pause", "wake up") end their turn
  after 800ms of trailing silence; longer speech keeps the full window
  (mid-thought pauses are why it's 1200). Pre-send stitching remains the
  net for a short opener of a longer sentence.
- **"Sleep go to sleep" reached the agent** (it faked a pause, ears stayed
  hot) — stacked repeats defeat the exact-phrase and segment passes. Fix:
  third pass in control_intent — all words within ONE intent's vocabulary
  AND a complete phrase present → control; any outside word disqualifies.
- **Onset-rejected spam**: --debug now prints the first rejection, then
  every 10th with a tally.

Re-verified after the fixes: 20 suites + ruff green; self-cut check at max
volume still 0 cuts (count + prose).

## Phases, each with a gate

### Phase 0 — the spike (go/no-go, ~1 focused session)

`scripts/spike_vpio.py`, standalone, no talk2me integration:

1. Record 10s via plain PortAudio mic while `say` plays through PortAudio
   → baseline echo RMS in capture.
2. Same, but capture via VPIO-enabled AVAudioEngine tap, TTS still
   through PortAudio → answers question (a) vs (b).
3. Same, TTS through an engine playerNode → the (b) number.
4. Log: echo attenuation dB in configs 2 and 3 vs 1, tap buffer format /
   rate / period, CPU %, whether ducking fired, whether a HUMAN voice
   over the TTS survives in the capture (record Nate counting over it).

**Gate:** ≥ ~20dB echo attenuation in config 2 or 3, human voice clearly
present in the cleaned capture, no crashes across 5 runs. Below the bar →
plan C ladder: tiny Swift helper binary running VPIO piping PCM to stdout
(more robust than PyObjC realtime taps, needs Xcode CLT at install), then
speexdsp/WebRTC-AEC3 in-process as the last resort. Do NOT grind past a
failed gate — that's how the last two weeks of failures happened.

### Phase 1 — VoiceProcessingMic (+Speaker if spike says (b))

- Implement behind hidden `--aec native`; PortAudio path untouched.
- Unit suite `test_vpio.py`: interface conformance against fakes, format
  conversion math, queue/mute semantics (real cancellation is untestable
  headless — that's what the field protocol is for).
- **Gate:** full existing suite green + a live smoke: session runs, hears
  Nate, transcribes correctly (STT accuracy spot-check vs PortAudio mic —
  VPIO's noise suppression must not degrade whisper).

### Phase 2 — integration

- Mode-resolution ladder + card copy + wizard note.
- Orchestrator simplifications (onset, gates) under native mode.
- Ducking configuration decision. Ctrl-T restart and clean-exit paths
  re-verified (engine teardown must not recreate the exit-crash class —
  hard-exit stays).
- Chores riding along: typed-input paste join bug ("heidk"), semaphore
  warning suppression if trivial.
- **Gate:** 19/20 suites green, Ctrl-T + Ctrl-C behave, no regression on
  headphones/phone/text modes.

### Phase 3 — the field protocol (the same tests that killed v2.3/v2.4)

All on speakers, `--debug`, transcripts kept. NORMAL volume, then MAX:

1. "Count 1 to 50" — silent throughout. **0 self-cuts.**
2. Prose answer (car engine / quantum) — silent. **0 self-cuts.**
3. Talk-over, couch voice, mid-sentence. **Cuts ≤1.5s, transcript = what
   Nate said, correct answer follows.**
4. Talk-over at MAX volume, couch voice. **Same bar — this is the test
   every previous version failed.**
5. Pause/"wake up" mid-task; task auto-resumes, never cancels.
6. 10 consecutive barge cuts in one session. **No wedge, no churn.**
7. Ctrl-C once. **Clean "bye.", no traceback, no segfault.**

**Gate:** all seven, both volumes. Any failure → fix → full protocol
reruns from the top.

## Explicitly NOT in v2.5

- **Enforcing voice-lock / TV immunity** — needs a stronger embedding
  model + live-score calibration (the observe-mode telemetry keeps
  collecting). Candidate for v2.6; don't let it creep in here.
- **Streaming neural TTS (Kokoro), Wispr, opencode adapter, NVIDIA
  brain** — separate tracks, untouched.
- **Linux AEC** — fallback ladder covers it; no bespoke Linux work.

## Risk register

| Risk | Mitigation |
|---|---|
| PyObjC realtime tap stutters under GIL pressure (whisper transcribing while capturing) | keep tap block minimal; convert off-thread; Phase 1 soak with whisper running; Swift-helper plan C |
| VPIO fights PortAudio's device claims / Bluetooth quirks | spike tests with BT connected; fallback ladder catches per-machine failure at startup |
| VPIO's processing hurts whisper accuracy | Phase 1 STT A/B; AGC/NS toggles if needed |
| Ducking makes TTS volume pump audibly | macOS 14 ducking config; worst case disable advanced ducking |
| macOS version matrix (10.15 baseline, ducking API 14+) | feature-detect at startup; anything missing → fallback ladder, card says so |
| A new deadlock class in engine teardown | stop/teardown off the event loop from day one; hard-exit stays; soak test in protocol #6 |

## Ship criteria, verbatim

Zero self-cuts at normal AND max volume; couch-voice talk-over cuts within
1.5s at both volumes; the full existing suite green; ten consecutive cuts
without a wedge; single-Ctrl-C clean exit. If any of these can't be met,
v2.5 does not ship — the public line stays on v2.2.0, and this document
gets a postmortem section like its predecessors.
